# 模型管理页面设计稿

## 1. 背景

本设计稿用于指导 `frontend-vite` 中新的模型管理页面实现。目标是在充分继承旧版 PySide 模型管理页面用户认知的前提下，将页面、拖拽排序与关联模态窗口迁移到 `frontend-vite` 的设计体系中。

本次设计基于以下已确认约束：

- 实现范围包含模型管理主页面，以及基础设置、任务设置、高级设置、模型选择四个关联模态窗口
- 新页面需参考 `frontend/Model/` 目录下的旧版页面代码与截图
- 除特别说明外，交互逻辑与用户认知保持与旧版一致
- 模型排序不再通过菜单中的“调整排序”完成，而是改为带手柄的拖拽排序
- 页面字体大小、间距、卡片样式参考 `frontend-vite` 已有设置页与设置卡片
- 模型列表必须使用 flow 容器，而非等比分隔的 grid 容器
- 保持旧版文案与图标，不新增自定义话术或额外语义
- 用户要求本次只继续设计与落文档，不进行 git 提交

## 2. 旧版页面基线

旧版 `frontend/Model/ModelPage.py` 的页面基线如下：

- 页面按模型类型分为四个分类卡片：
  - 预设模型
  - 自定义 Google 模型
  - 自定义 OpenAI 模型
  - 自定义 Anthropic 模型
- 每个分类卡片顶部包含品牌色竖条、标题、描述；自定义分类右上角有“新增”按钮
- 分类内容区使用流式排列的模型按钮，而不是严格等宽网格
- 每个模型按钮点击后展开菜单，菜单包含：
  - 激活模型
  - 基础设置
  - 任务设置
  - 高级设置
  - 预设模型为“重置模型”，自定义模型为“删除模型”
  - 旧版还有“调整排序”子菜单，但本次会移除
- 当前激活模型使用更强的主按钮视觉态
- 基础设置、任务设置、高级设置均为独立大尺寸模态窗口
- 基础设置中的“获取”会再打开模型选择模态窗口

## 3. 设计目标与非目标

### 3.1 设计目标

- 保持旧版用户对“模型分类、模型切换、模型编辑入口”的认知连续性
- 将视觉语言统一到 `frontend-vite` 当前设置页体系
- 将模型组内排序迁移为拖拽手柄交互
- 保持页面信息结构清晰，避免把复杂编辑逻辑塞回主页面
- 保持状态来源单一，避免主页面与模态窗口维护多份真相

### 3.2 非目标

- 不重构为左右分栏内联详情页
- 不将主要操作改成显式常驻按钮条
- 不引入跨分类拖拽与模型类型迁移
- 不新增新的字段、文案、提示分类或复杂筛选能力
- 不改变旧版即时写回的交互心智

## 4. 方案比较与结论

### 4.1 方案 A：保守迁移

保留四个分类卡片、保留点击模型名称打开操作菜单、保留独立模态窗口编辑，仅将排序改成拖拽手柄，并将整体视觉统一到 `frontend-vite` 设计体系。

优点：

- 最贴近旧版用户认知
- 需求改动点集中，回归范围可控
- 与用户已确认边界完全一致

缺点：

- 主页面的效率提升有限
- 菜单式操作仍然偏保守

### 4.2 方案 B：半显式操作

保留分类卡片，但在模型项上直接露出部分高频操作，其他动作进入菜单。

优点：

- 常用操作更快

缺点：

- 会显著改变旧版认知
- 模型项密度变高，视觉更嘈杂

### 4.3 方案 C：单页详情驱动

左侧为模型列表，右侧或底部为内联编辑区，弱化模态窗口。

优点：

- 操作集中

缺点：

- 偏离旧版认知最明显
- 与“模态窗口页面”要求冲突

### 4.4 选择结论

采用 **方案 A：保守迁移**。

选择原因：

- 用户已明确要求保留旧版交互逻辑与认知
- 用户已确认模型项交互采用保守迁移
- 用户已确认配置编辑仍使用模态窗口页面
- 拖拽排序是唯一需要主动改变的主页面心智入口

## 5. 主页面设计

### 5.1 页面总体结构

模型管理页继续使用纵向页面结构，包含四个分类卡片，顺序保持不变：

1. 预设模型
2. 自定义 Google 模型
3. 自定义 OpenAI 模型
4. 自定义 Anthropic 模型

每个分类卡片包含：

- 左侧品牌色竖条
- 标题
- 描述
- 可选的“新增”按钮
- 使用 flow 容器承载的模型项列表

页面不使用等比分隔 grid。模型项使用自适应换行的 flow 容器排列，以保留旧版浏览体验与组内扫描方式。

### 5.2 分类卡片视觉

分类卡片应复用 `frontend-vite` 的卡片视觉基线：

- 卡片表面、边界、阴影、圆角使用现有 `Card` 体系
- 标题与描述的字号、间距、文本层级参考 `setting-card-row`
- 分类卡片之间的纵向间距与设置页卡片列表一致
- 卡片头部信息密度向设置页靠拢，但不改变旧版“标题 + 描述 + 操作”的基本布局

品牌色只用于分类标题左侧竖向强调条，不扩散到整张卡片背景，避免破坏当前渲染层设计系统的一致性。

### 5.3 模型项结构

每个模型项由两个紧邻元素组成：

- 左侧：拖拽手柄按钮
- 右侧：模型名称下拉按钮

其中：

- 拖拽手柄只负责发起排序拖拽
- 模型名称按钮只负责显示当前模型名并打开操作菜单
- 当前激活模型保留更强的按钮视觉态，以延续旧版主按钮认知
- 非激活模型保持普通按钮态

模型项不直接显式展示“激活 / 设置 / 删除”等多个常驻按钮，避免改变旧版操作路径。

### 5.4 模型菜单

模型菜单保留旧版动作与顺序，只移除排序相关动作：

1. 激活模型
2. 基础设置
3. 任务设置
4. 高级设置
5. 重置模型 / 删除模型

菜单文案与图标继续沿用旧版已有语义，不新增新的命名。

旧版“调整排序”子菜单彻底移除。排序入口统一收束为拖拽手柄。

## 6. 模态窗口设计

### 6.1 总体原则

基础设置、任务设置、高级设置、模型选择都继续实现为独立模态窗口页面。

设计约束如下：

- 不改成内联展开
- 不改成侧边抽屉
- 不在主页面直接承载编辑表单
- 模态窗口继续承担“进入细节编辑”的职责
- 模态窗口关闭动作继续保持单主按钮“关闭”
- 视觉使用 `frontend-vite` 的卡片、输入框、选择器与对话框体系

### 6.2 基础设置模态窗口

基础设置模态窗口继续是大尺寸滚动表单窗口，字段与旧版一致。

字段结构：

- 模型名称
- 接口地址
- 接口密钥
- 模型标识
- 思考等级

交互要求：

- `模型名称` 与 `接口地址` 为单行设置卡片，右侧是输入框
- `接口密钥` 保持为大文本框，不改成标签或列表型控件
- `模型标识` 卡片继续包含三个按钮：
  - 输入
  - 获取
  - 测试
- 点击“获取”后打开模型选择模态窗口
- 点击“测试”后继续走统一测试状态与 toast 反馈
- 任务忙碌时禁用测试按钮，延续旧版行为
- 所有字段继续即时写回，不增加“保存”按钮

### 6.3 任务设置模态窗口

任务设置模态窗口继续保持极简卡片列表，不增加新的信息层级。

字段结构：

- 输入 Token 限制
- 输出 Token 限制
- 并发任务数限制
- 每分钟请求数限制 (RPM)

交互要求：

- 每项使用数字输入控件
- 每次修改后即时写回
- 不新增推荐值、智能提示或折叠说明

### 6.4 高级设置模态窗口

高级设置模态窗口继续保留旧版的两类内容：

- 生成参数
- 自定义请求配置

生成参数保留四项：

- `top_p`
- `temperature`
- `presence_penalty`
- `frequency_penalty`

每项继续保持：

- 一个启用开关
- 一个滑条
- 一个当前值显示

行为要求：

- 开关关闭时恢复默认值并写回
- 开关打开后显示滑条和值
- 滑条值变更后即时写回

请求配置保留两组：

- 自定义请求头
- 自定义请求体

行为要求：

- 每组保留启用开关
- 每组下方保留大文本框用于 JSON 输入
- 空值允许保存为空对象
- 输入不是 JSON 对象时提示格式错误
- 失焦时再给明显提示，避免输入过程中频繁打断
- 保持旧版“请谨慎设置，错误的值可能导致结果异常或者请求报错”语义

### 6.5 模型选择模态窗口

模型选择模态窗口继续保持中尺寸列表式对话框，不升级成复杂表格或多维筛选器。

结构保持为：

- 标题
- 描述
- 搜索过滤框
- 加载中区域
- 模型列表

交互要求：

- 打开后延迟发起模型列表加载，先让模态窗口完成渲染
- 加载中显示明确的等待反馈
- 获取失败时使用 warning toast 提示
- 过滤框实时筛选模型列表
- 点击模型后立即写回 `model_id`
- 写回成功后直接关闭窗口

## 7. 状态流与数据边界

### 7.1 单一状态源

模型页及其关联模态窗口只认一份权威状态，即模型页面快照。

约束如下：

- 主页面基于最新模型快照分组渲染四个分类
- 模态窗口打开时读取当前选中模型对应的快照数据
- 模态窗口内字段修改通过统一更新入口写回
- 主页面在模态窗口关闭后重新同步最新快照
- 不允许主页面与模态窗口长期持有相互独立的可编辑副本

### 7.2 页面状态职责

主页面状态负责：

- 加载模型快照
- 按模型类型分组
- 记录当前激活模型
- 控制各类模态窗口开关
- 处理新增、激活、删除、重置、排序动作

模态窗口状态负责：

- 展示单个模型的当前字段
- 派发字段更新
- 管理局部交互状态，如加载中、测试中、筛选词

局部状态仅用于交互，不作为模型数据的长期真相来源。

### 7.3 拖拽排序规则

拖拽排序严格限制在单个分类组内。

明确禁止：

- 预设模型拖到自定义分类
- Google 模型拖到 OpenAI 分类
- OpenAI 模型拖到 Anthropic 分类
- 通过拖拽改变模型类型

用户已明确确认，本次排序只允许组内调整。

### 7.4 拖拽持久化策略

拖拽排序采用“本地乐观更新 + 服务端快照兜底”的策略。

推荐流程：

1. 用户拖动组内模型项
2. 前端在当前分类中乐观更新顺序
3. 拖拽结束后提交该分类最终的有序模型 id 列表
4. 服务端返回最新模型页面快照
5. 前端使用服务端快照重新覆盖本地状态

失败策略：

- 排序接口失败时回退到拖拽前顺序
- 使用 toast 提示排序失败

本设计不建议沿用旧版 `MOVE_UP / MOVE_DOWN / MOVE_TOP / MOVE_BOTTOM` 的菜单式操作模型作为拖拽落地接口语义。拖拽更适合提交“最终顺序”，这样也能减少前端将拖拽结果翻译成离散命令的复杂度。

## 8. 错误处理与反馈

### 8.1 保留旧版反馈的场景

以下反馈必须原样保留旧版语义：

- 删除分类中的最后一个模型时，提示 warning toast 并阻止删除
- 删除模型前显示确认对话
- 重置预设模型前显示确认对话
- 重置成功后显示成功 toast
- 测试模型通过统一状态与 toast 反馈结果
- 获取模型列表失败时显示 warning toast
- JSON 格式错误时显示明确提示

### 8.2 新增反馈场景

由于引入拖拽排序，本次新增一个反馈场景：

- 拖拽排序持久化失败时，页面回退顺序并提示失败 toast

这个新增反馈不改变旧版认知，只是对新交互的必要补充。

### 8.3 忙碌态约束

涉及外部请求或可能引起冲突的动作，需要遵守旧版忙碌态约束：

- 基础设置中的“测试”按钮在任务忙碌时禁用
- 如果某些操作受 API 状态约束，前端应基于统一状态源禁用对应交互
- 忙碌态的来源必须唯一，不允许某个弹窗自行维护第二套“是否可测试”判断

## 9. `frontend-vite` 落位建议

建议在 `frontend-vite` 中以下述结构承载本次实现：

```text
src/renderer/pages/model-page/
  page.tsx
  model-page.css
  types.ts
  use-model-page-state.ts
  components/
    model-category-card.tsx
    model-item-chip.tsx
  dialogs/
    model-basic-settings-dialog.tsx
    model-task-settings-dialog.tsx
    model-advanced-settings-dialog.tsx
    model-selector-dialog.tsx
```

落位原则：

- 模型页专属组件优先留在 `pages/model-page/` 内
- 先不提升到 `widgets/`，避免形成伪共享层
- 页面状态集中在 `use-model-page-state.ts`
- 对话框作为模型页专属交互，优先留在 `dialogs/` 目录

理由：

- 当前这些组件与模型页状态、文案和交互耦合较深
- 暂时没有稳定的跨页面复用边界
- 先靠近使用点实现，再根据后续复用情况决定是否上提

## 10. 验证要求

本次实现完成后，至少需要回归以下场景：

- 四个分类卡片是否按正确顺序展示
- 分类标题、描述与新增按钮是否与旧版语义一致
- 模型项是否以 flow 方式换行排列，而不是等比分栏 grid
- 当前激活模型是否具有更强按钮态
- 点击模型名称是否仍打开操作菜单
- 菜单是否保留激活、基础设置、任务设置、高级设置、重置/删除
- 旧版排序菜单是否已经完全移除
- 拖拽手柄是否只允许组内排序

- 新增自定义模型是否仍走正确分类

- 模型选择弹窗加载、过滤、点击回写是否正常

### 10.2 工程验证要求

若本设计进入实现阶段，应至少执行：

- `frontend-vite` 相关静态检查与必要构建验证

- 若新增纯函数或状态转换逻辑，优先补相应测试

本设计阶段不执行实现，也不提交 git。

## 11. 最终结论

本次模型管理页面采用“保守迁移”方案：

- 保留四分类卡片结构
- 保留点击模型名称打开菜单的旧认知
- 将组内排序改为拖拽手柄
- 保留四个独立模态窗口页面
- 视觉与密度统一到 `frontend-vite` 设置页设计体系
- 保持旧版文案、图标与即时写回逻辑

这样既满足新前端的一致性要求，也最大程度避免用户在迁移后重新学习模型管理工作流。
