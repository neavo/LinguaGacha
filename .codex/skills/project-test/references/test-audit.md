# 测试整理与审计流程

只在任务要求检查、整理、优化、合并、清理测试用例，或要求检查业务文件与测试文件映射关系时读取本文件。

## 审计目标

测试整理的目标不是保留更多用例或追求覆盖率数字，而是让保留下来的测试更准确、更稳定、更能证明当前业务行为。

优先级：
1. 行为证明价值
2. 映射关系清晰
3. 边界隔离合理
4. 执行效率可接受
5. 覆盖率数字

## 映射审计

先建立业务文件与测试文件的一一映射表，再改具体用例。

检查项：
- 每个业务文件是否有唯一测试文件。
- 每个测试文件是否只对应一个主业务文件。
- 文件名和路径是否遵循项目现有映射规则。
- 是否存在同一业务文件被多个测试文件覆盖。
- 是否存在一个测试文件混测多个不相关业务文件。
- 公用夹具、测试工具、测试数据 builder 是否只提供准备能力，没有承载具体业务断言。

处理规则：
- 一对多：合并到唯一测试文件，删除或重命名多余测试文件。
- 多对一：拆回各自业务文件的对应测试文件；只保留真实跨业务流程作为集成测试。
- 多对多：先按业务文件切分断言，再决定是否保留独立集成测试。
- 无对应测试文件：需要新增有效行为测试时创建；没有值得测试的当前行为时不要为了映射完整而凑文件。

## 逐用例审查

对每个测试用例回答这四个问题：
1. 它证明哪条当前有效的业务行为、边界或错误分支？
2. 它断言的是公开结果，还是内部实现偶然细节？
3. 这个场景在当前业务逻辑中是否真的可触达？
4. 如果删除它，是否会丢失一个真实风险的防护？

结论只有四类：
- 保留：行为价值明确，断言稳定。
- 修改：目标有价值，但写法白盒、脆弱或准备成本过高。
- 合并：单独存在价值弱，但可并入更完整的业务场景。
- 清理：孤儿、凑数、过时、实际不可触达，或只服务覆盖率数字。

## 清理低价值用例

这些用例应优先修改、合并或删除：
- 只断言对象存在、长度非零、函数被调用。
- 测试名和数据都说不清业务意义。
- 只覆盖内部缓存、私有字段、临时状态或 mock 调用细节。
- 依赖已经废弃的入口、模式、配置或历史兼容分支。
- 理论上可达，但当前产品路径、输入约束或上游校验已经保证基本不会触达。
- 为了覆盖率硬造异常、空值、组合输入，而业务域没有对应需求。
- 多个测试只换一个无语义值，实际证明同一件事。

不要只用“有没有引用”判断孤儿用例。仍有引用但业务路径已经不会触达的测试，也可能是低价值测试。

## 业务透明重构

为了提升测试效果，可以对目标模块做小范围业务透明重构。

允许：
- 提取纯函数，让核心规则脱离 IO、时间、线程、网络。
- 注入 clock、id generator、transport、repository 等外部边界。
- 拆分过大的函数，使公开行为和副作用边界更清楚。
- 收口重复分支，让测试聚焦真实业务差异。
- 增加轻量 value object 或明确返回结果，替代观察内部状态。

禁止：
- 为了测试改变业务语义、默认值、错误处理或持久化结果。
- 为了更好 mock 而暴露不该公开的内部细节。
- 把真实业务流程拆成一堆只服务测试的人工入口。
- 用重构掩盖旧测试失败，而不解释行为变化是否真实需要。

## 推荐执行顺序

1. 列出目标业务文件和现有测试文件。
2. 建立并修正一一映射关系。
3. 按对应测试文件逐个审查测试用例。
4. 先清理明显孤儿、凑数和过时用例。
5. 再把有价值但写法差的用例改成行为断言。
6. 如测试很难写，判断是否需要业务透明重构。
7. 跑目标测试；再按改动范围扩大验证。
8. 交付时说明保留、修改、合并、清理了哪些测试类别。

## 交付自检

- 映射关系没有一对多、多对一、多对多。
- 每个保留用例都能说清当前业务价值。
- 清理不是按“无引用”机械删除，而是按当前业务可触达性与风险价值判断。
- 覆盖率下降不一定是失败；如果删除的是低价值用例，应说明原因。
- 业务透明重构有明确测试收益，并通过原有行为验证兜底。
