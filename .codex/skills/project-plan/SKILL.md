# 项目计划

## 核心判断

先判断当前对话模式，再决定计划产物放在哪里：

- 如果最新开发者上下文显示 `Collaboration Mode: Plan`，可以创建正式实施计划，并使用 `update_plan` 或当前环境提供的计划工具跟踪步骤。
- 如果当前处于非计划模式，不在回复中输出正式实施计划；只做需求深挖、上下文澄清、前置理解草稿和事实取证，并把需要沉淀的阶段性计划材料维护到 `PLAN.md`。
- `PLAN.md` 是非计划模式下唯一允许修改的计划材料文件，可记录前置理解、已确认事实、缺口、待确认事项、草案步骤和清理范围；它不属于长期权威文档，也不能替代计划模式中的正式计划。
- 非计划模式下允许读取项目文档、源码、测试、配置和本地工具输出进行计划前取证；取证结果只能用于澄清需求、暴露缺口、维护 `PLAN.md` 或提示用户切换计划模式。
- 除维护 `PLAN.md` 外，非计划模式不得修改代码、长期文档或其它任务资产，不得跑验证，不得调用会改变项目状态的执行类工具。
- 不要在默认模式下用普通回复冒充计划，也不要借编辑 `PLAN.md` 顺手实施方案。

## 创建流程

在计划模式中，结合用户请求、已有上下文和仓库约束创建计划：

1. 明确产品目标、目标用户、核心场景、成功判据、交付物、边界条件和不能做的事。
2. 先执行意图与证据提取，并把提取结果汇入「意图与事实前置理解摘要」；简单任务可以压缩，复杂任务必须完整展开。
3. 再收集足够的产品事实和工程事实制定具体实施步骤；上下文不足时，主动到产品文档、设计文档、仓库文档、现有代码、测试、配置和本地工具输出中取证。
4. 确认产品意图是否影响交互、状态归属、数据模型、跨层载荷、唯一写入口、验证入口和文档同步范围。
5. 若仍存在会影响产品目标、用户体验或工程路径，且无法从上下文或项目现状推断的信息，先向用户提问，暂停正式计划。
6. 使用 `update_plan` 工具或当前环境提供的创建计划工具生成计划；计划第一项必须承载意图与事实前置理解，并保持后续状态可跟踪。
7. 把清理施工现场列为独立步骤，并明确列出需要清理的目标、范围、处理方式和保留理由。

## 意图与事实前置理解摘要

正式计划必须以「意图与事实前置理解摘要」开头，把隐式理解变成可检查的产品意图和工程事实；它属于计划正文。按任务复杂度调整长度：

| 任务复杂度 | 摘要要求 |
| --- | --- |
| 简单任务 | 用 3 到 5 行确认产品目标、核心场景、边界和验证入口。 |
| 中等任务 | 输出完整摘要，每项保持简短。 |
| 复杂任务 | 输出完整摘要，并列出需求来源、产品取向、已确认路径和未决问题。 |
| 信息不足任务 | 先输出摘要和缺口，再提问；不得直接生成正式计划。 |

摘要至少覆盖：

- `产品目标`：用户真正希望获得的改善、效率、质量、体验或风险降低效果，用非技术语言表达。
- `目标用户 / 使用场景`：谁在什么上下文、流程节点或压力条件下使用这项能力。
- `当前痛点`：现状造成的困惑、低效、体验断裂、维护负担或风险。
- `理想体验`：改完后用户应该如何感知结果、反馈、流程或行为变化。
- `成功判据`：用可观察行为、验收点、测试、文档事实或指标判断目标达成。
- `已确认需求点`：对话中用户明确提出、确认或修正过的需求。
- `已确认实现路径`：已经讨论并确认过的做法、技术路线、范围边界和不采用的路径。
- `禁止事项 / 不做范围`：用户明确排除的内容，以及仓库规则禁止的行为。
- `涉及项目现实`：会影响实现的架构层级、权威文档、文件候选、状态归属、验证入口。
- `仍缺信息`：无法从上下文和项目现状推断，且会影响产品目标、体验判断或实现路径的问题。
- `提问清单`：只问真正阻塞计划质量的问题；能靠取证解决的不要问用户。

## 意图与证据提取

生成摘要前，必须把可见上下文和必要取证结果转换成结构化证据。不要只声称已经阅读、回顾或理解；每个会影响计划方向的判断都要落到摘要字段、缺口门禁或步骤依据中。

| 证据来源 | 提取内容 | 写入位置 |
| --- | --- | --- |
| 用户消息 | 明确需求、排除项、确认过的方向、验收口径、用户纠偏 | `已确认需求点`、`禁止事项 / 不做范围`、`成功判据` |
| `PRODUCT.md` / `DESIGN.md` | 产品目标、目标用户、核心场景、体验边界、设计取向 | `产品目标`、`目标用户 / 使用场景`、`理想体验`、`成功判据` |
| 项目长期文档 | 架构层级、状态归属、唯一写入口、公开协议、验证入口、文档同步范围 | `涉及项目现实`、`已确认实现路径` |
| 源码 / 测试 / 配置 / 脚本 | 真实调用链、类型形状、状态拥有者、现有测试覆盖、命令入口 | `涉及项目现实`、`仍缺信息` |
| 外部官方资料 | 当前依赖、框架、CLI、云服务或协议事实 | `涉及项目现实`、`仍缺信息` |

提取规则：

- 每条关键结论都写成“结论 + 来源或依据”，来源可以是用户消息、权威文档、代码位置、测试入口或工具输出。
- 没有证据支撑的摘要字段必须标为 `仍缺信息`，并说明它会影响产品方向、体验判断、实现路径或验证方式中的哪一项。
- 会改变交互、数据模型、状态归属、跨层载荷或验证方式的缺口，必须进入缺口门禁处理。
- 只对影响计划质量的证据做结构化提取；普通背景信息、无关搜索结果和大段文件内容不要搬进计划。
- 正式计划中的每个关键实施步骤都应该能追溯到提取出的产品意图、需求事实、路径承诺或项目约束；追溯不了的步骤通常属于猜测，应删除、取证或提问。

## 事实获取

计划中的产品意图和工程事实不只来自对话上下文。制定计划前必须按任务风险逐级补齐事实，直到足以写出“基于事实”的步骤：

1. 先使用对话中已经给出的需求、约束、错误信息和用户确认过的事实。
2. 若任务影响用户可见行为、交互、信息架构、文案或产品规则，先读取 `PRODUCT.md`、`DESIGN.md` 和相关页面 / 功能现状，确认产品目标与体验边界。
3. 再读取项目内的权威文档、相关源码、测试、配置、脚本和历史约定，基于实际文件确认现状，避免凭目录名推断。
4. 若仍缺少关键事实，运行只读或低风险的本地检查命令获取证据，例如定位引用、查看类型定义、读取测试入口或确认工具脚本。
5. 涉及第三方库、框架、CLI、云服务或普通网络信息时，使用项目规定的文档检索或网络检索工具确认最新事实。
6. 只有当事实无法从本地项目或可靠资料中获得，且不同答案会显著改变计划时，才向用户提问；提问前先说明已经确认了什么、仍缺什么。

凡是会决定产品目标、用户场景、成功判据、计划主路径、文件范围、状态归属、唯一写入口、跨层载荷或验证入口的事实，都必须在正式计划成形前完成取证。不要把这些事前准备包装成实施步骤；已确认事实应汇入计划开头的「意图与事实前置理解摘要」。

只有实施过程中才可能暴露、或需要改动后才能验证的事项，才可以写入计划中的检查、验证或回看步骤。

## 缺口门禁

如果产品意图或工程事实不足以决定实现路径，不要把猜测写成执行步骤。按以下门禁处理：

| 情况 | 处理方式 |
| --- | --- |
| 产品目标、目标用户或成功判据缺失，且会影响方案方向 | 先从用户对话、`PRODUCT.md`、`DESIGN.md`、现有行为中取证；仍无法确认时提问，暂停正式计划。 |
| 目标用户 / 使用场景不清，但任务只影响内部重构 | 写明“用户可见行为保持一致”的依据，继续计划。 |
| 存在多个产品取向，且会导向不同交互、数据模型或验证方式 | 先列出取舍点和影响，向用户确认后再生成正式计划。 |
| 缺口可从仓库文档、源码、测试、配置中确认 | 先自行取证，不问用户。 |
| 缺口可从已有对话合理推断，且风险低 | 写明推断依据，继续计划。 |
| 缺口无法从上下文或项目现状推断，且会改变方案 | 先问用户，暂停正式计划。 |
| 缺口不影响当前计划主路径 | 放入“非阻塞假设 / 后续确认”，继续计划。 |

提问前先说明已经确认了什么、仍缺什么、为什么这个缺口会改变计划。

## 取证展示粒度

取证结果要透明但克制：

| 内容 | 展示方式 |
| --- | --- |
| 读过哪些权威文档 / 源码 / 测试 / 配置 | 简短列出。 |
| 确认到的关键产品意图和工程事实 | 写入计划开头的「意图与事实前置理解摘要」。 |
| 影响路径选择的证据 | 在复杂任务步骤的 `依据` 字段中引用。 |
| 大段文件内容 / 普通搜索结果 | 不展示。 |
| 与计划无关的旁支发现 | 不展示；除非构成风险。 |

## 计划质量

计划必须立足产品意图和项目现实，避免想象或猜测：

- 每个步骤都应该能被执行者直接落地，避免“完善逻辑”“处理问题”这类空泛表述。
- 每个步骤都必须写出 `目的`、`实施方案` 和 `阶段成果`，不能只用一句标题或简单编号带过。
- 复杂任务的每个步骤必须额外写出 `依据`，说明它来自哪些产品目标、用户场景、成功判据、已确认需求、实现路径或项目事实。
- 不确定的事实必须先调查，再实现；不要把未经确认的设计、文件位置、状态归属或验证入口当作既定事实。
- 计划中的关键实现步骤必须能追溯到产品目标、用户场景、成功判据、已确认需求和项目事实；若只能写出“可能需要”“预计会”，说明事实仍不足，应继续取证。
- 任何会影响体验、交互、文案、数据形状或状态归属的步骤，都必须回答“它服务哪个用户场景里的哪个成功判据”。
- 步骤粒度按工作量划分：一个步骤应有清晰产出，但不要细到单个无意义编辑动作。
- 跨层任务要显式列出阅读文档、定位权威来源、实现、验证和文档同步。
- 验证步骤要写出具体命令或检查对象；如果验证条件未知，先列出确认验证入口的步骤。
- 计划应覆盖收尾动作：回看 diff、确认命名与边界、按目标和范围清理废弃资源、说明未执行或失败的验证。

## 步骤描述格式

计划中的每个步骤都必须使用同一结构，确保执行者无需重新猜测意图：

```text
1. 步骤名称
   - 依据：复杂任务必填，说明来自产品目标 A、用户场景 B、成功判据 C、已确认需求 D、实现路径 E、项目事实 F。
   - 目的：说明这一步要消除什么不确定性、建立什么前提，或完成什么可交付变化。
   - 实施方案：
     1. 写出第一项可执行动作。
     2. 写出第二项可执行动作。
     3. 必要时继续列出检查对象、文件、命令或决策点。
   - 阶段成果：说明这一步完成后会得到什么产品判断、工程事实、用户可感知行为、代码改动、测试结果、文档更新或决策结论。
```

规则：

- `实施方案` 至少包含 2 个可执行子项；复杂步骤可以扩展到 5 个左右，但不要堆砌无意义动作。
- 简单和中等任务可以省略逐步 `依据` 字段，但复杂任务、争议路径和多方案取舍必须写出。
- `目的` 要解释为什么这一步存在，不要复述步骤标题。
- `阶段成果` 必须可检查，例如“确认权威写入口是某文件中的某类”“得到待修改文件清单”“通过某条验证命令”。
- 如果某一步只是等待用户确认，也要写明确认项、确认方式和确认后进入哪个后续步骤。

## 推荐结构

优先使用 5 到 8 个步骤。简单任务可以更短，跨层任务可以更长；输出时按“步骤描述格式”展开每一步即可，示例只保留骨架：

```text
1. 意图与事实前置理解摘要
2. 梳理产品目标、用户场景、成功判据和不做范围
3. 提取并补齐产品、设计、工程和工具输出中的关键证据
4. 补齐仍缺失的关键事实，必要时查官方资料或向用户提问
5. 定位权威状态、唯一写入口和验证入口
6. 实施与目标一致的闭环改动
7. 清理施工现场
8. 执行验证
9. 回看 diff 并同步文档
```

## 清理施工现场

清理步骤不能省略，也不能只写原则。计划中必须明确列出本次需要清理的目标和范围；没有对应清理项时，也要写明“已检查，无需清理”的依据。

清理计划建议使用以下结构：

```text
- 清理目标：具体到文件、目录、类型、函数、样式类、本地化键、测试资产或文档段落。
- 清理范围：说明只检查或修改哪些模块、页面、测试、配置或文档，不触碰哪些无关区域。
- 处理方式：删除、合并、迁移、重命名、保留兼容层或仅记录。
- 保留理由：若暂不清理，说明仍被谁使用，以及后续移除条件。
```

根据任务实际影响，至少检查这些目标类型：

- 代码资源：不再使用的代码分支、变量、函数、类、类型、常量、导出和导入。
- 前端资源：不再使用的组件、样式类、设计 token、页面局部状态、路由入口和图标资源。
- 文案资源：废弃文案、本地化键、错误信息、提示语和文档碎片。
- 测试资源：过期测试、夹具、快照、mock、脚本参数和临时测试文件。
- 配置资源：废弃配置项、环境变量、构建脚本、CLI 参数和工具入口。

边界要求：

- 不做与本任务无关的大范围重构；发现无关脏改时只记录，不擅自回滚。
- 保留必要兼容层时，必须说明兼容对象、当前调用方和后续移除条件。
- 清理范围必须和已确认事实一致，不能凭猜测删除看似无用的资源。
