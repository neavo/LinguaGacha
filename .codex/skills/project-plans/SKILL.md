# Project Plans

## 核心判断

先检查当前对话是否处于计划模式：

- 如果最新开发者上下文显示 `Collaboration Mode: Plan`，继续创建计划。
- 如果当前不是计划模式，停止规划，并用简短中文提示用户先切换到计划模式。
- 不要在默认模式下用普通回复冒充计划，也不要开始改代码、跑验证或调用执行类工具。

建议提示语：

```text
🧭 当前还不是计划模式，请先切换到计划模式，我再为你创建一份可执行计划。
```

## 创建流程

在计划模式中，结合用户请求、已有上下文和仓库约束创建计划：

1. 明确最终目标、交付物、边界条件和不能做的事。
2. 读取或引用已经在上下文中可用的事实；缺少关键信息时，把“获取事实”列为靠前步骤。
3. 确认改动涉及的层级、状态拥有者、唯一写入口、跨层载荷和验证入口。
4. 使用 `update_plan` 工具或当前环境提供的创建计划工具生成计划，并保持后续状态可跟踪。
5. 把清理施工现场列为独立步骤，覆盖此次变更后不再使用的代码、变量、方法、文本、本地化条目、测试资产和文档碎片。

## 计划质量

计划必须立足现实，而不是想象或猜测：

- 每个步骤都应该能被执行者直接落地，避免“完善逻辑”“处理问题”这类空泛表述。
- 不确定的事实必须先调查，再实现；不要把未经确认的设计当作既定事实。
- 步骤粒度按工作量划分：一个步骤应有清晰产出，但不要细到单个无意义编辑动作。
- 跨层任务要显式列出阅读文档、定位权威来源、实现、验证和文档同步。
- 验证步骤要写出具体命令或检查对象；如果验证条件未知，先列出确认验证入口的步骤。
- 计划应覆盖收尾动作：回看 diff、确认命名与边界、清理废弃资源、说明未执行或失败的验证。

## 推荐结构

优先使用 5 到 8 个步骤。简单任务可以更短，跨层任务可以更长，但每一步都必须有明确目标。

```text
1. 梳理目标与约束
2. 阅读必要文档和现有实现
3. 定位权威状态与唯一写入口
4. 实施最小闭环改动
5. 清理施工现场
6. 执行验证
7. 回看 diff 并同步文档
```

## 清理施工现场

清理步骤不能省略。根据任务实际影响检查：

- 删除或整理不再使用的代码分支、变量、函数、类型、样式和组件。
- 移除废弃文案、本地化键、测试夹具、脚本参数和临时文件。
- 保留必要兼容层时，说明它仍被谁使用，以及后续移除条件。
- 不做与本任务无关的大范围重构；发现无关脏改时只记录，不擅自回滚。
